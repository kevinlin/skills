SAP Integration — Payload Design Blockers
Vendor cannot build the API calls without BSC SAP team answering these first
IDGap / RiskWhy CriticalSeverityWorkstreamOwner
R-01 Sales org / dist channel / division / order type not specified for VA01 calls
Required fields in every VA01 payload for both the accrual posting (FR_M19) and credit note (FR_M25). Without them the API call cannot be constructed at all.
 Vendor cannot size or design WS6 integration without these values. Blocks all VA01 development. Critical WS6 BSC SAP Team
R-02 Bundle pricing condition type unnamed
Section 3.1 puts bundle sync in scope but no condition type is named. Vendor cannot write the VK11 payload for bundle pricing without this.
 Blocks VK11 bundle integration development entirely. Critical WS5 BSC SAP Team
R-03 VK11 validity from/to date rules not specified for AOP rate changes
<Project> must send validity dates in every VK11 call. If wrong or unspecified, in-flight dealer orders get repriced silently with no error returned by SAP.
 Silent mispricing risk at every AOP changeover. No detection mechanism. Critical WS5 BSC SAP Team
R-04 FR_G19 special price — create new condition record or override existing?
Determines the API verb (POST vs PATCH) and the access sequence configuration in SAP. These are completely different implementations.
 Vendor cannot implement FR_G19 without BSC confirming the intended SAP behaviour. Critical WS5 BSC SAP Team
AMB-06 SAP condition type: ZJHD (NFR-I03) vs ZJHP (FR_Q14) — conflicting specs
FR_Q14 references SAP condition type ZJHP; FR_Q23 references ZJHD. NFR-I03 lists only ZJHD as in-scope. It is unclear whether ZJHP is a distinct type, a typo for ZJHD, or an obsolete condition type.
 SAP pricing records written with the wrong condition type will silently produce incorrect prices in MyOrders and billing. Critical data integrity risk requiring a single authoritative answer. High WS5 BSC SAP Team
GAP-04 SAP near-real-time price sync SLA (<5 min) — FeatureSpec P0, no NFR exists
FeatureSpec §SAP-Integration lists near-real-time SAP price sync with a <5-minute SLA as a P0 requirement. No FR or NFR captures this. Achieving <5 min requires either a JARVIS push trigger or high-frequency polling — neither is specified. Conflicts with the T+1 ADL batch architecture.
 If BSC expects sub-5-minute price propagation, the current ADL nightly batch architecture cannot support it. Requires a real-time integration design that is not currently scoped. High WS5 BSC / Vendor
AMB-07 SAP condition type ZMAP in FR_G15 — real type or documentation placeholder?
ZMAP is mentioned as an example in FR_G15. It is unclear whether this is a live BSC SAP condition type, a placeholder, or a deprecated type. Designing integrations against a placeholder condition type produces unusable artefacts.
 Integration artefacts built against ZMAP may be invalid. BSC SAP team must confirm the authoritative condition type list. Medium WS5 BSC SAP Team

WS6 Rebate Accounting — Unresolved Integration Boundaries
SAP internal postings are a black box — but the handoff point and payload design are not fully defined
IDGap / RiskWhy CriticalSeverityWorkstreamOwner
CTR-09 Accrual trigger — manual user action (FR_M15) vs automated schedule (FeatureSpec) — direct contradiction
FR_M15 in requirements FINAL.md states accrual is triggered by "pressing a button." FeatureSpec §Accrual states "automated accrual calculation on a defined schedule — not manual." These are diametrically opposed.
 Determines the entire accrual engine architecture, scheduling infrastructure, audit trail requirements, and user training scope. Cannot be designed or built until BSC resolves this at source. Critical WS6 BSC Finance
G-02 VA01 vs FB50 contradiction in FR_M19
FR_M19 says "journal entry (equivalent to VA01)." VA01 is a sales document; FB50 is a direct FI journal entry. These are mutually exclusive — completely different API calls, different payload structures, different account logic.
 Determines the entire technical integration architecture. All other WS6 design decisions depend on this answer. Critical WS6 BSC SAP Team
G-01 Accrual debit side GL account unnamed
FR_M18_SAP names only the credit side (AR control account 110010). The debit side — a P&L expense or cost account — is named nowhere in the requirements.
 SAP cannot be configured to post the accrual without both sides of the entry. Critical WS6 BSC SAP Team
G-03 Accrual reversal mechanism absent — ledger doubling risk
No requirement mentions accrual reversal, netting, or clearing between monthly accrual postings (FR_M19) and the period-end credit note (FR_M25). If both post independently to the same account, the balance doubles.
 Ledger integrity risk. BSC Finance must confirm whether settlement triggers reversal of monthly accruals. Critical WS6 BSC Finance
G-11 Credit note target sub-ledger unnamed — FR_M25 completely silent
FR_M18_SAP covers accrual only. FR_M25 names no customer code. Case A (dummy rebate customer): clears 110010 but dealer receives nothing in their AR. Case B (real dealer customer): dealer AR credited but 110010 never clears.
 Vendor cannot construct a valid VA01 payload without a target customer code. The rebate cycle cannot close financially without this resolved. Critical WS6 BSC SAP Team
G-05 1-step vs 2-step credit note process unclear
Standard SAP credit note processing is VA01 (credit memo request) → VF01 (billing document). Only VA01 is mentioned. If 2-step, the VF01 billing step is missing from the process model and must be assigned to either the vendor or BSC SAP team.
 If 2-step and unaccounted for, the credit note never posts to AR until VF01 runs — a missing step in the BPMN. High WS6 BSC SAP Team
G-06 Dealer payment method not described
FR_M26 outputs a confirmation report to the dealer. No requirement describes how the dealer receives financial value — cash, applied against next invoice, or AR offset.
 Without this, BSC Finance cannot confirm the rebate cycle is complete from a ledger perspective. High WS6 BSC Finance
G-07 SAP document type code for credit note not specified
VA01 can create KA (credit memo request), RE (returns), ZCR (custom), etc. Each triggers different account determination logic. No document type code is named in the requirements.
 Mandatory field in the VA01 API call. BSC SAP team must confirm which type is configured for rebate credit notes. High WS6 BSC SAP Team
G-12 Pooled dummy customer obscures individual dealer accrual liability
All monthly accruals for all dealers pool to the same dummy customer / reconciliation account 110010. SAP sub-ledger cannot show how much of the balance belongs to each dealer. Finance relies entirely on <Project>'s rebate tracker for dealer-level visibility.
 If <Project> and SAP diverge, there is no named mechanism to detect or resolve the discrepancy. Missing internal control. High WS6 BSC Finance
G-13 Order reversal mid-accrual period — no handling defined
If a sales order is reversed in SAP after the accrual has been calculated against it, <Project>'s accrual balance overstates the liability. No requirement defines how <Project> detects or corrects this via the T+1 ADL feed.
 Accrual balance will be overstated for any reversed orders until the next settlement cycle. No correction mechanism exists. High WS6 BSC Finance
G-14 Cross-period returns — no requirement for accrual adjustment
A return from a prior settlement period creates a negative accrual that spans closed accounting periods. Requirements do not address how <Project> handles the accrual and settlement adjustments for out-of-period returns.
 Cross-period returns will create unresolved balances in closed periods. Finance must define the accounting treatment. High WS6 BSC Finance
G-15 Post-settlement returns — credit note already issued, dealer has been paid
If a dealer returns goods after the rebate settlement credit note has been issued and applied, the rebate has been overpaid. No requirement defines the recovery mechanism: chargeback, deduction from next period, or write-off.
 Post-settlement returns with no recovery mechanism result in unrecoverable overpayments. BSC Finance must define the policy. High WS6 BSC Finance
G-17 Goods return + rebate credit note — compounding double-loss with no detection mechanism
When a dealer returns goods, BSC issues a returns credit note (DR Revenue / CR Dealer AR) reversing the sale. Independently, <Project> fires a rebate credit note (DR 110010 / CR Dealer AR) at settlement against an accrual base that was never corrected for the return. The dealer receives double benefit — goods returned and rebate paid on those goods — with no system-level link between the two credit notes. The correcting journal on return (DR 110010 / CR Rebate Expense) does not exist anywhere in the current process model.
 BSC incurs a double commercial loss on every return where a rebate credit note has been issued or is pending for the same period. Neither SAP nor <Project> raises an exception. Exposure scales with return frequency and average rebate rate. High WS6 BSC Finance
G-18 FR_M01 returns exclusion — SAP order type and order reason combinations not registered
FR_M01 requires returns to be excluded from the Sales A accrual base. The specific SAP order type and order reason code combinations that identify return documents in BSC Japan are never defined in the requirements. Without these values, <Project>'s ADL ingestion logic cannot filter out returns at import — all return transactions will be included in the accrual base as positive contributions.
 The entire premise of FR_M01 — a clean Sales A data set free of returns — cannot be realised. Run rate, target achievement, accrual, and settlement figures will all be overstated. BSC SAP team must provide the complete list of order type + order reason combinations before the accrual engine can be trusted. High WS6 BSC SAP Team
G-19 Allowance / hospital rebate accounting entirely unanalysed — SAP posting mechanics unknown
Two distinct rebate programme architectures exist: dealer rebates (FR_M series, sell-in, monthly accrual → VA01 credit note) and Allowance rebates (FR_Q series, sell-out / hospital / primary market, allowance budget → post-discount ADL master sync). The entire accounting gap analysis covers dealer rebates only. Whether Allowance settlements trigger a SAP accrual posting, a VA01 credit note, or no SAP posting at all is completely open.
 If Allowance settlements carry any SAP financial impact, all gap questions G-01 through G-18 apply to a second, entirely unmodelled posting flow. Proceeding without confirming this is an untested assumption that could double WS6 integration scope. High WS6 BSC Finance
AMB-08 FR_M26 settlement acceptance by inaction — trigger, mechanism, and legal enforceability undefined
FR_M26 states that customer silence during a "configured period" constitutes acceptance. The period start trigger is undefined. The "contact" mechanism (email? in-app? portal?) is undefined. Legal enforceability under Japanese commercial law of silence-as-acceptance is not addressed.
 Settlement process cannot be designed. Legal risk in Japan where explicit acceptance may be required for financial instruments. High WS6 BSC Finance
G-08 SAP field for <Project> contract number not named
FR_G06 identifies the contract number as the end-to-end key but does not name the SAP field (PO number BSTNK, reference XBLNR, or custom). Without this, the contract number cannot be passed in the API payload.
 Breaks the traceability chain from <Project> contract to SAP posting. Prevents cross-system reconciliation. Medium WS6 BSC SAP Team
G-09 Accrual vs settlement variance posting not described
Monthly accruals are run-rate estimates. Actual settlement will differ. No requirement addresses how the variance is posted or reversed when actuals are confirmed.
 If actual rebate > cumulative accruals, an additional posting is needed. If actual < accruals, a reversal is needed. Neither case is modelled. Medium WS6 BSC Finance
G-10 No finance reconciliation step between SAP and <Project>
FP&A is named as an approver (FR_W02) but there is no requirement for finance to reconcile posted SAP accruals to <Project> figures. A key internal control is missing from scope.
 No check that the SAP ledger matches the <Project> rebate tracker. Medium WS6 BSC Finance
G-16 Month-end cut-off — transactions submitted but not approved before period close
No requirement defines how <Project> handles accrual proposals that are submitted but not yet approved when the accounting period closes. Should they be included in the closing accrual, held to the next period, or escalated?
 Unapproved accruals at period-end create an undefined ledger state. Finance cannot lock the period until this is resolved by policy. Medium WS6 BSC Finance
MFR-11 Accrual reversal workflow — FeatureSpec P1, no FR exists
FeatureSpec §Accrual lists user-initiated accrual reversal as a P1 feature. No FR exists. SAP accrual reversals (credit memos, FI reversals) have specific transaction requirements. This is required for error correction and period-close adjustments.
 Without a defined reversal workflow, error correction requires manual SAP intervention outside <Project> — audit trail gap and governance risk. Medium WS6 Vendor
MFR-12 Settlement dispute tracking — FeatureSpec P1, no FR exists
FeatureSpec §Settlement lists dispute tracking as P1. No FR exists. Customer challenges to settlement amounts require a state machine, ownership assignment, resolution workflow, and audit trail. Absence means disputes are handled offline.
 Settlement disputes handled offline defeat the purpose of a managed settlement process. No audit trail. Finance exposure undefined. Medium WS6 Vendor
G-04 Standalone credit note — confirmed correct by design
Rebate credit notes do not need to reference a specific billing document. Rebates are retrospective obligations spanning many invoices. A standalone VA01 is standard SAP practice. The O2C cycle being complete when settlement posts is expected and by design.
 Not a blocking gap. Removed as a design concern — superseded by G-11. Resolved WS6 BSC SAP Team

CTR-06 Concurrent user load — RFP states 1,200; NFR-P02 states 200 APAC / 100 Japan — 6× discrepancy
RFP §5 states 1,200 concurrent users (APAC-wide). NFR-P02 states 200 APAC / 100 Japan. Infrastructure sizing, load balancing, and AWS instance type selection are directly determined by this figure. A 1,200-user requirement is approximately 6× the infrastructure cost of a 200-user requirement.
 Bidding against the wrong figure results in either a significantly underpriced infrastructure (if 1,200 is correct) or an overpriced bid (if 200 is correct). Must be resolved before architecture is designed. Critical Cross BSC
